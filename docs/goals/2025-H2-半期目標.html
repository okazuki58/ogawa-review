2025-H2-半期目標

2025H2 半期目標

期間: 2025年9月〜2026年2月（6ヶ月）

現在のグレード: G2（作業者）

目標グレード: G3（自走可能）

目標概要

G3昇進を目指し、前回評価（2025H1）の課題を踏まえた3つの目標を設定。

  • Be hungry: A評価 → S評価へ
  • Be an owner: B評価 → A評価へ
  • Be sustainable: B評価 → A評価へ

目標1: Be hungry（貪欲になれ）- 前回A評価 → S評価へ

半期目標

設計力を向上させ、学びを週1回以上恒常的にアウトプットする

具体的な成果指標

  • 設計関連の技術書を2冊以上読破（DB設計、アーキテクチャパターンなど）
  • 学んだ設計パターンを実プロジェクトで3回以上適用
  • AWSの体系的な学習を行う（AWS認定資格取得を目指す、難しい場合でも体系的な学習を継続）
  • 技術ブログ/Qiita記事を隔週で投稿 → 半期で12本以上
  • 詳細な技術ドキュメントを2本作成（Confluence/Google Slide/動画解説など、形式は自由）
  • AIを使った設計の最適化手法をドキュメント化
  • 細かいナレッジ共有（Slackなど）は気づいたときに随時実施

狙い

前回「アウトプットを恒常的に」との指摘を受け、週1回以上の発信を仕組み化。また、AIが苦手とする設計力を深めることで技術力を高める。

目標2: Be an owner（当事者たれ）- 前回B評価 → A評価へ

半期目標

新規事業の成功に向けて、チーム全体の生産性を高める情報基盤を構築する

具体的な成果指標

  • 開発ドキュメント（アーキテクチャ図、API仕様、セットアップ手順など）を月1回以上更新
  • 週次で進捗・課題・学びをまとめて事業部長・社長と共有（macloggerと連携） → 半期で24回
  • GitHub Issues/Projectsでタスク管理を可視化（進捗を誰でも見られる状態に）
  • CoderabbitなどのAIレビューツールを導入し、自動レビューの仕組みを構築
  • 新機能開発時、既存コードとの整合性を確認するチェックリストを作成・運用

狙い

前回「チーム全体の生産性向上のための情報蓄積やコミュニケーション」が課題。一人開発でも、情報を可視化・共有することでチーム全体に貢献できる。

目標3: Be sustainable（継続的であれ）- 前回B評価 → A評価へ

半期目標

属人化を解消し、自分以外の誰でも開発を引き継げる状態にする

具体的な成果指標

  • 全ての主要機能にREADME/ドキュメントを整備（最低10機能）
  • 環境構築を標準化・自動化し、誰でも30分以内に開発を始められる状態にする
  • 手段: Docker化、セットアップスクリプト、READMEの整備など
  • 狙い: 属人化防止、新メンバーがすぐに参加できる
  • 繰り返し作業（デプロイ、テスト、データ投入など）を3つ以上自動化
  • トラブルシューティングガイドを作成（過去の障害とその解決方法を記録）
  • 技術的判断の根拠をドキュメント化（なぜこの技術を選んだか、なぜこの設計にしたか）

狙い

前回「自分以外の誰でも同じことをできるか」が課題。ドキュメント化と自動化を徹底し、属人化を解消する。また、技術的判断の根拠を残すことで、将来の引き継ぎやスケールに備える。

この目標がG3昇進につながる理由

G3の要件: 自発的に作業を切り出して進められる

上記3つの目標は、いずれも「指示されずに自ら課題を見つけて改善する」姿勢を示すものです。

  • Be hungry S評価: 技術的な深掘りと発信力 → 自発的な学習・成長
  • Be an owner A評価: チーム全体の生産性向上 → 自分の担当範囲を超えた貢献
  • Be sustainable A評価: 仕組み化・ドキュメント化 → 組織の継続性への貢献

これらが達成できれば、G3相当の「自発的に作業を切り出して進められる」レベルに到達できます。
