# 目標設定スキル

## 概要

このスキルは、半期目標の設定を支援します。会社のMission/Value、グレード制度、評価基準に基づいて、半期→月次→週次へ階層的に目標を分解します。

**会社の半期定義**:
- **上半期（H1）**: 3月〜8月（6ヶ月）
- **下半期（H2）**: 9月〜2月（6ヶ月）

**主な機能**:
- 階層的目標設定（半期→月次→週次）
- 評価項目との対応明記（各目標がどの評価項目に対応するか明確化）
- タスクチェック形式 `[ ]` での目標記載

## 使用タイミング

- 半期の目標設定を行う時
- 月次・週次の目標に分解する時
- 目標が評価基準のどの項目に対応するか確認したい時

## 主要機能

### 1. 階層的目標設定

#### 1-1. 半期目標の設定

ユーザーの現在のグレード、役割、過去の実績を踏まえて、以下の観点から**半期目標**を提案：

- **Be hungry**: 学習・成長・高みを目指す目標
  - 評価項目1: 貪欲に新しいものを勉強したか
  - 評価項目2: 貪欲に仕事に取り組んだか
- **Be an owner**: 当事者意識・責任感を示す目標
  - 評価項目1: 締め切り意識を持てたか
  - 評価項目2: チーム全体の生産性に寄与したか
- **Be sustainable**: 継続性・効率化・健康を意識した目標
  - 評価項目1: 属人化しないように仕組み化するようになにかやれたか
  - 評価項目2: 日常業務に対して改善する動きができたか

**プロセス**:
1. ユーザーの現状を深掘り（担当業務、技術スタック、課題感、キャリアイメージ）
2. グレード要件を確認
3. 3つのValueに沿った半期目標を提案
4. **各目標には必ず対応する評価項目を明記する**
5. SMARTな目標になるよう調整

#### 1-2. 月次目標への分解

半期目標を達成可能な**月次目標**に分解：

```
半期目標: Railsのアーキテクチャパターンを学び、設計力を向上させる

↓ 分解

月次目標（6ヶ月分全部作成）:
- 1ヶ月目（9月）: 「Everyday Rails」を読み、基本パターンを理解
- 2ヶ月目（10月）: 学んだパターンをチームに共有、フィードバック収集
- 3ヶ月目（11月）: 小規模機能の設計ドキュメントを作成
- 4ヶ月目（12月）: 中規模機能の設計・実装
- 5ヶ月目（1月）: 設計レビューの実施
- 6ヶ月目（2月）: 半期の成果まとめ・発表
```

#### 1-3. 週次目標への分解

月次目標をさらに**週次目標**に分解（各月を4週に分解、計24週分）：

```
月次目標（9月）: 「Everyday Rails」を読み、基本パターンを理解

↓ 分解

週次目標:
- Week 1（9月第1週）: Chapter 1-3を読む
- Week 2（9月第2週）: Chapter 4-6を読む、サンプルコード写経
- Week 3（9月第3週）: Chapter 7-9を読む
- Week 4（9月第4週）: 学んだことをまとめ、勉強会資料作成

※同様に10月、11月、12月、1月、2月の各月も4週に分解（計24週）
```

### 2. 評価基準の確認

- グレード昇進の可能性チェック
- 評価ランク（S/A/B/C/D）の予測
- 給与への影響の説明

## ワークフロー

### 目標設定時（階層化対応）

**CRITICAL**: 目標設定時は以下の順序で**必ず**実行すること

#### ステップ1: プロフィール情報の確認

```
1. my-profile が利用可能かチェック（プロジェクトルート/my-profile/）
   - 利用可能 → my-profile/SKILL.md を読み込み
   - references/*.md も読み込み（background, current-role, strengths-weaknesses, career-goals）
   
2. my-profile がない場合
   → ユーザーに profile-builder skill でプロフィール作成を促す
   「目標設定の前に、まずプロフィールを作成しましょう。
    profile-builder skill を使って、30分-1時間かけて高解像度のプロフィールを作成してください。
    プロフィールは目標設定の土台となる重要なステップです。」
   
3. プロフィール情報がある場合
   → ユーザーへの質問をスキップし、プロフィールの情報を使用
```

#### ステップ2: 評価基準の確認

```
1. references/grades.md でグレード要件を参照
2. references/values.md でValue基準を参照
3. 前回評価があれば プロジェクトルート/resources/original-sheets/配下のCSVを確認
```

#### ステップ3: 半期目標の提案

```
1. 3つのValueに沿った半期目標を提案
   - Be hungry: 
     - 評価項目1: 貪欲に新しいものを勉強したか
     - 評価項目2: 貪欲に仕事に取り組んだか
   - Be an owner: 
     - 評価項目1: 締め切り意識を持てたか
     - 評価項目2: チーム全体の生産性に寄与したか
   - Be sustainable: 
     - 評価項目1: 属人化しないように仕組み化するようになにかやれたか
     - 評価項目2: 日常業務に対して改善する動きができたか

2. **各目標には必ず対応する評価項目を明記する**
   フォーマット:
   【評価項目: Be hungry - 1. 貪欲に新しいものを勉強したか】

3. **全ての目標に必ず `[ ]` をつける**
   - 後から振り返りでチェックできるようにする
   - 達成したら `[x]` に変更できる
   フォーマット:
   - [ ] 具体的な目標内容

4. SMARTな目標になるよう調整
   - Specific（具体的）
   - Measurable（測定可能）
   - Achievable（達成可能）
   - Relevant（関連性）
   - Time-bound（期限）

5. 成果指標を明確に設定
```

#### ステップ4: 自動的に階層分解（MUST）

**重要**: 半期目標を提案したら、**自動的に**以下の分解を行うこと。ユーザーに「分解してください」と言われるのを待ってはいけない。

```
1. 月次目標に分解（半期分全部 = 6ヶ月分）
   - 各月の具体的なアクション
   - マイルストーン設定
   - 例: 2025H2なら9月、10月、11月、12月、1月、2月の6ヶ月分
   
2. 週次目標に分解（半期分全部 = 約24週分）
   - 各月を4-5週に分解
   - 各週の具体的なタスク
   - 週次の成果物
   - 例: 9月Week1-4、10月Week1-4、...、2月Week1-4
   
3. ファイルに保存（プロジェクトルート/goals/配下に保存）
   - プロジェクトルート/goals/YYYY-HX-半期目標.md（1ファイル）
   - プロジェクトルート/goals/YYYY-HX-月次目標.md（1ファイルに6ヶ月分まとめて記載）
   - プロジェクトルート/goals/YYYY-HX-週次目標.md（1ファイルに半期分全部（約24週）まとめて記載）
   
   **重要**: .claude/skills/performance-review/goals/ではなく、プロジェクトルートのgoals/に保存すること
```

#### ステップ5: 確認と調整

```
1. 生成した目標をユーザーに提示
2. 調整が必要か確認
3. 必要に応じて修正
```

### 目標の調整・修正

目標を生成した後、ユーザーが調整したい場合の対応：

```
1. どの部分を調整したいか確認
   - 「どの目標を調整したいですか？」
   - 「高すぎる/低すぎる/不要な目標はありますか？」

2. よくある調整パターン
   
   a) 目標が高すぎる
   - 「記事12本→6本に減らしたい」
   - 「技術書2冊→1冊に減らしたい」
   → 数値目標を調整し、月次・週次も再計算
   
   b) 目標を追加したい
   - 「XXXも目標に加えたい」
   → 適切な評価項目を確認し、[ ] 形式で追加
   
   c) 目標を削除したい
   - 「この目標は今期は不要」
   → 該当目標を削除し、月次・週次から除外
   
   d) 評価項目が不明確
   - 「これはどの評価項目に対応しますか？」
   → 評価項目を明記・調整

3. 調整後の再生成
   - 半期目標を修正
   - 月次目標を再計算（6ヶ月分）
   - 週次目標を再計算（約24週分）
   - ファイルを上書き保存
```

## 重要な評価原則

1. **事実ベース**: すべての評価は具体的な事実・エピソードに基づく（ログから自動抽出可能）
2. **3つのValue**: Be hungry / Be an owner / Be sustainable の観点を常に意識
3. **評価項目の明記（必須）**: 各目標には必ず対応する評価項目を明記する
   - Be hungry: 評価項目1（勉強）／評価項目2（仕事への取り組み）
   - Be an owner: 評価項目1（締め切り意識）／評価項目2（チーム生産性）
   - Be sustainable: 評価項目1（仕組み化）／評価項目2（業務改善）
4. **グレード要件**: 現在のグレードの期待値を明確に理解する
5. **階層的目標**: 半期→月次→週次に分解し、達成可能にする

## 評価ランクの基準

- **S (想定を大きく超えている)**: 給与アップ
- **A (期待値ちょい上)**: 給与アップ
- **B (期待値通り)**: 給与ステイ
- **C (期待値に達していない)**: 前回C以下の場合、給与ダウン
- **D (出来ているとはいえない)**: 給与ダウン

## 目標記載フォーマット

**重要**: 各目標には必ず対応する評価項目を明記すること

### フォーマット例

**重要**: 全ての目標に `[ ]` をつけて、後から振り返りでチェックできるようにする

```markdown
## Be hungry（貪欲になれ）

### 【評価項目1: 貪欲に新しいものを勉強したか】

- [ ] 設計関連の技術書を2冊以上読破（DB設計、アーキテクチャパターンなど）
- [ ] 学んだ設計パターンを実プロジェクトで3回以上適用
- [ ] AWSの体系的な学習を行う（AWS認定資格取得を目指す）

### 【評価項目2: 貪欲に仕事に取り組んだか】

- [ ] 技術ブログ/Qiita記事を隔週で投稿 → **半期で12本以上**
- [ ] 詳細な技術ドキュメントを2本作成（Confluence/Google Slide/動画解説など）
- [ ] AIを使った設計の最適化手法をドキュメント化

## Be an owner（当事者たれ）

### 【評価項目1: 締め切り意識を持てたか】

- [ ] すべてのタスクを期限内に完了（やむを得ない場合は事前相談）
- [ ] 週次報告を欠かさず実施

### 【評価項目2: チーム全体の生産性に寄与したか】

- [ ] 月1回のチーム勉強会を主催
- [ ] コードレビューを積極的に実施（週5件以上）
- [ ] 新メンバーのオンボーディング支援

## Be sustainable（継続的であれ）

### 【評価項目1: 属人化しないように仕組み化するようになにかやれたか】

- [ ] 開発手順書を5本以上作成
- [ ] 既存の属人化タスクを2つ以上ドキュメント化・引き継ぎ

### 【評価項目2: 日常業務に対して改善する動きができたか】

- [ ] CI/CDパイプラインの改善（ビルド時間20%短縮）
- [ ] 繰り返し作業の自動化スクリプトを3つ以上作成
```

## 使用例

**ユーザー**: 「今期の目標を立てたい」

→ Claude: my-profile/ をチェック
  - **my-profile/ がある場合**:
    - my-profile/ を読み込み（現在のグレード、目標グレード、前回評価などを確認）
    - 現在のグレード要件を確認し、目標グレードを目指せる具体的な目標を提案
    - **各目標に評価項目との対応を明記**
    - 半期→月次→週次に分解
  
  - **my-profile/ がない場合**:
    - 「目標設定の前に、まずプロフィールを作成しましょう。
       profile-builder skill を使って、30分-1時間かけて高解像度のプロフィールを作成してください。
       プロフィールは目標設定の土台となる重要なステップです。」

**ユーザー**: 「目標の達成状況を確認したい」
→ maclogger-review スキルを使用してください（このスキルは目標設定専門です）
